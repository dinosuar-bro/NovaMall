# 测试与验收方案

## 1. 原则

- 先定义失败用例，再实现功能；
- 测试真实边界，不以大量 Mock 替代数据库约束；
- 每个开发阶段形成可独立运行的最小闭环；
- 未通过阶段质量门禁，不进入下一阶段；
- 所有性能和并发结论来自真实执行结果。

## 2. 测试层级

### 静态验证

- TypeScript 严格类型检查；
- ESLint；
- 格式和 Markdown 链接检查；
- SQL 格式与迁移顺序检查；
- 禁止 `any` 和未解释的类型断言。

### 单元测试

- Service 业务状态机；
- 权限和资源归属判断；
- 金额转换与展示；
- 错误映射；
- React 表单、状态组件和纯交互逻辑。

### API 与数据库集成测试

使用隔离的 MySQL 测试数据库，真实执行迁移、存储过程、触发器、视图和事务。覆盖正常、越权、冲突、回滚和并发。

### E2E 测试

通过浏览器验证最关键用户旅程，不追求覆盖所有管理表格：

1. 注册、登录和地址；
2. 申请开店并由管理员批准；
3. 店主创建并上架商品；
4. 会员加购多个商品、统一结算和支付；
5. 店主发货，会员确认收货；
6. 管理员只读查看审计日志和有效销量 Top 10 数据库证据。

## 3. 核心测试矩阵

| 领域 | 正常用例 | 边界与失败用例 |
|---|---|---|
| 注册登录 | 注册、登录、退出 | 重复用户名、弱密码、错误密码、禁用账号 |
| 地址 | CRUD、默认地址 | 越权地址、并发设置默认、订单快照不变 |
| 开店 | 提交、批准、拒绝、重提 | 重复待审核、重复审核 |
| 商品 | CRUD、上架、库存、图片 | 非店主访问、负价格、非法状态、伪装图片 |
| 搜索 | 分类、关键词、排序 | 空关键词、中文短词、停用分类、下架商品 |
| 购物车 | 增删改、商品列表 | 重复添加、零数量、归档商品、库存变化 |
| 结算 | 统一结算、快照、清车 | 空购物车、库存不足、并发抢购、重复 Token |
| 支付 | 模拟支付 | 重复支付、取消后支付、金额不一致 |
| 履约 | 发货、确认收货 | 非店主发货、重复发货、越权确认 |
| 退款 | 申请、批准、拒绝 | 非待发货、重复申请、非店主审核、库存恢复 |
| 排行 | 有效销量 Top 10 | 取消/退款排除、并列稳定排序 |
| 审计 | 自动记录、查询 | 回滚不留成功日志、敏感字段脱敏 |
| 加密 | 加密、授权解密 | 错误密钥、不同 IV、非授权接口泄露 |

## 4. 数据库专项验收

详细实验见 `docs/database/experiment-plan.md`。至少包含：

- 存储过程输入、输出、回滚和幂等；
- 触发器价格历史和审计；
- 视图结果一致性；
- 索引前后 `EXPLAIN ANALYZE`；
- 两连接并发抢最后库存；
- 窗口函数 Top 10；
- 全文检索与 LIKE 对比；
- AES 密文和授权解密验证。

## 5. API 验收

- 响应结构符合 `docs/api.md`；
- Zod 拒绝非法请求；
- 401、403、404、409 和 500 语义正确；
- 请求编号贯穿响应和日志；
- BIGINT ID 以字符串输出；
- 金额以两位小数字符串输出；
- 分页总数和边界正确。

## 6. 安全验收

- 无明文密码和手机号落库；
- 日志、审计和错误不泄露敏感数据；
- SQL 注入、CSRF、跨角色和会员资源越权测试通过；
- Session 登录时轮换，退出后不可复用；
- 禁用账号的旧 Session 立即失效；
- 上传目录无法执行文件。

## 7. Docker 验收

- 从空卷执行一次性启动；
- 数据库迁移成功且健康检查通过；
- 前端、API 和图片路径可访问；
- 重启容器后数据库和图片仍存在；
- 缺少必要 Secret 时后端拒绝启动并给出安全错误；
- 测试环境和开发环境数据库隔离。

## 8. 阶段质量门禁

### 最终验收命令

```bash
CI=true pnpm lint
CI=true pnpm typecheck
CI=true pnpm test
TEST_DATABASE_URL='mysql://novamall:novamall_test_password@127.0.0.1:3308/novamall_test' CI=true pnpm test:integration
env -u CI ./node_modules/.bin/playwright test
CI=true pnpm build
docker compose config
git diff --check
```

数据库集成测试前需要启动测试 MySQL 并执行迁移：

```bash
docker compose -f docker-compose.test.yml up -d mysql-test
TEST_DATABASE_URL='mysql://novamall:novamall_test_password@127.0.0.1:3308/novamall_test' pnpm db:test:migrate
```

阶段 6 自动化覆盖建立在 Stage 5 已完成能力之上：共享合同、API 单元、认证/商户入驻/商品目录/结算履约 API 与数据库集成、Web 角色页与路由守卫、Playwright 浏览器 E2E。E2E 会通过 `seed-demo` 服务准备 `demo_owner` 和 `demo_admin` 演示账号，并覆盖会员提交开店申请、管理员批准、会员进入店主后台，以及管理员创建分类、店主发布带图商品、会员按关键词搜索商品的最小链路；Stage 6 已验证加购、地址、结算、支付、店主发货、会员确认收货和管理员数据库证据入口，同时补齐 `docs/evidence/database/` 的真实 SQL、执行计划、计时和并发证据。

当前 Docker 验收补充：

- 主库映射 `127.0.0.1:3307`，测试库映射 `127.0.0.1:3308`；
- 后端上传目录使用 `uploads-data` 卷持久化；
- 前端对历史缺失商品图片显示统一占位图；
- 变更前端静态资源或后端镜像内容后，需要使用 `docker compose up --build -d` 而不是只重启容器。

每阶段结束必须满足：

1. 阶段需求、接口、表结构和验收文档已更新；
2. 新增测试通过；
3. 既有测试无回归；
4. TypeScript 类型检查通过；
5. Lint 通过；
6. 前后端构建通过；
7. 涉及数据库时，迁移可从空库执行且专项测试通过；
8. Docker 健康检查通过；
9. `git diff --check` 通过；
10. 实际结果已报告，未运行项明确标注。

## 9. 最终验收场景

使用店主、一个普通会员和一个管理员完成现场演示：

1. 管理员创建分类；
2. 会员申请开店，管理员批准；
3. 店主发布有库存商品；
4. 普通会员按分类和关键词找到商品；
5. 加入多个商品到购物车并一次结算；
6. 展示一个总订单和履约子订单；
7. 模拟支付；
8. 一个子订单发货并确认收货；
9. 展示 Top 10、审计日志和 9 项数据库技术证据。

退款只保留数据库专项测试路径，不要求 Stage 5 前端页面覆盖。

## 10. Stage 6 证据验收

Stage 6 不新增业务功能主线，重点验收证据是否足以支撑课程报告和答辩。正式证据文件位于 `docs/evidence/database/`：

- `01-存储过程.md`：`sp_checkout_cart` 成功、失败回滚、幂等和 OUT 参数；
- `02-触发器.md`：触发器直接 DML 验证；
- `03-视图.md`：视图与基础表一致性；
- `04-索引优化.md`：索引查询的 `EXPLAIN ANALYZE`、计时和数据规模；
- `05-事务与并发控制.md`：两连接并发、事务回滚和无负库存；
- `06-窗口函数.md`：窗口函数 Top 10 与并列稳定排序；
- `07-审计日志.md`：审计日志脱敏、成功记录和失败不伪记；
- `08-全文检索.md`：ngram FULLTEXT 与 LIKE 的执行计划、计时和语义差异；
- `09-AES数据加密.md`：AES 密文、授权解密、错误密钥和订单快照验证。

报告中的性能数字、并发结论和安全结论只能引用这些证据文件。每个文件需包含环境、数据规模、命令或 SQL、实际输出、结论和限制；未运行的实验必须明确标注“未验证”，不能使用预计结果替代。
