# 商品图片显示异常问题修复总结

## 问题概述

商品目录中出现部分商品图片正常、部分商品图片无法显示或显示异常的问题。最初表现为部分商品卡片顶部为空白区域，后续又发现有一张商品图片显示为纯色红块。

该问题并不是单一的前端样式问题，而是由历史测试数据、上传目录配置和图片兜底逻辑共同导致。

## 问题原因

### 1. 后端上传目录不一致

后端运行时曾把上传文件写入 `/app/apps/api/uploads`，但 Docker 持久化卷挂载在 `/app/uploads`。

因此，数据库中保存的部分图片路径虽然存在，但容器重建后对应文件不在正确目录中，访问时会返回 404，前端商品卡片就会出现图片空白。

### 2. 历史测试图片尺寸过小

早期 E2E 测试上传过 1x1 PNG。浏览器认为该图片加载成功，因此不会触发 `onError`，但图片本身几乎不可见，视觉上类似空白图片。

### 3. 测试图片本身是纯色图

后续为了避免 1x1 图片，将 E2E 测试图改成了 64x64 PNG，但该图片本身是纯色红块。浏览器会正常加载它，所以前端无法把它识别为加载失败，最终表现为商品图区域显示红色块。

## 修复流程

### 1. 统一上传目录

后端新增 `UPLOAD_ROOT` 配置，并在 Docker 中设置为 `/app/uploads`。静态资源访问统一通过该目录提供，避免容器重建后上传文件丢失。

### 2. 迁移历史上传文件

将旧目录 `/app/apps/api/uploads` 中已有的图片复制到 `/app/uploads` 持久化卷中，恢复历史图片访问能力。

### 3. 增加前端图片兜底

商品图片加载失败时，前端会自动切换为统一占位图。

同时增加图片尺寸判断：如果图片加载成功，但自然尺寸小于 `32x32`，也会切换为占位图，避免 1x1 图片造成视觉空白。

### 4. 增加后端上传校验

后端上传接口对 PNG 图片增加最小尺寸校验。小于 `32x32` 的图片会返回 `IMAGE_TOO_SMALL`，避免未来继续产生不可见商品图片。

### 5. 生成并接入新占位图

使用生图模型生成新的商品占位图，并保存为：

```text
apps/web/public/product-placeholder.png
```

前端商品卡片使用该图片作为统一兜底图。

### 6. 修复 E2E 测试数据

将 E2E 中的商品图片上传从手写测试 PNG 改为上传真实占位图文件，避免测试继续生成纯色商品图。

### 7. 修复当前历史数据

在 Docker 上传卷中定位已知测试 PNG 的 hash，仅替换这些历史测试图文件为新的占位图，避免误伤其它真实图片。

## 验证结果

本次修复后执行了以下验证：

- `CI=true pnpm typecheck` 通过
- `CI=true pnpm lint` 通过
- `CI=true pnpm test` 通过
- `TEST_DATABASE_URL='mysql://novamall:novamall_test_password@127.0.0.1:3308/novamall_test' CI=true pnpm test:integration` 通过
- `CI=true pnpm build` 通过
- `env -u CI ./node_modules/.bin/playwright test` 通过，结果为 `5 passed`

页面探测确认，商品目录中的异常商品图已经正常加载为新的 `900x675` 占位图。

## 修复结论

该问题本质是上传文件持久化路径不一致、历史测试图片质量不合格、前端兜底逻辑不足共同造成的。

修复后，系统从运行环境、后端校验、前端展示和测试数据四个层面形成闭环。后续不会再因为缺失图片、过小图片或测试纯色图片导致商品目录显示异常。
