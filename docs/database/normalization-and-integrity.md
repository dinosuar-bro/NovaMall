# 数据库范式与完整性约束

## 1. 范式目标

NovaMall 的业务主数据以第三范式（3NF）为目标，在历史快照、金额快照和审计日志中进行受控反范式。反范式字段必须由事务或触发器维护，不能依赖前端计算。

## 2. 第一范式（1NF）

每个字段只保存一个不可再分的业务值：

- 用户角色不以逗号字符串保存在 `users`，而拆为 `roles` 和 `user_roles`；
- 订单商品不以 JSON 数组保存在订单主表，而拆为 `order_items`；
- 地址拆为省、市、区和详细地址；
- 店主资料与商品兼容关联、分类与商品均通过外键表达。

`audit_logs.old_data/new_data` 使用 JSON 是审计事件快照，不参与业务关系运算，因此属于有意的半结构化日志，不代表业务主表违反 1NF。

## 3. 第二范式（2NF）

数据库中的大部分表使用单列代理主键，不存在非主属性只依赖复合主键一部分的问题。

`user_roles` 使用 `(user_id, role_id)` 复合主键，其附加字段 `granted_at` 和 `granted_by` 依赖完整的“用户获得某角色”事实，而不只依赖其中一列，因此满足 2NF。

`cart_items` 使用代理主键，并额外约束 `(user_id, product_id)` 唯一；数量依赖完整的用户与商品组合。

## 4. 第三范式（3NF）

- 角色名称只存于 `roles`，不在 `users` 重复保存；
- 店主资料只存于 `shops`，商品后续按共享商品池经营；
- 分类资料只存于 `categories`，商品不重复保存分类名称；
- 支付信息独立于总订单，退款信息独立于子订单；
- 店主身份通过 `user_roles` 表达，店主资料归属通过 `shops.owner_user_id` 表达。

非主属性直接依赖各表候选键，不通过其他非主属性产生传递依赖。

## 5. 受控反范式

### 订单商品快照

`order_items` 重复保存商品名称、图片路径和成交单价。其目的是保留交易发生时的事实，避免商品改名、换图或调价污染历史订单。

### 地址和手机号快照

`master_orders` 保存收件人、地址和加密手机号快照。会员修改或删除地址后，历史订单仍可解释。

### 金额快照

`line_amount`、`subtotal_amount` 和 `total_amount` 可由明细推导，但作为交易快照保存，以减少展示时重复计算并固定成交事实。存储过程在同一事务中计算并写入，测试验证三层金额一致。

### 审计 JSON

审计日志保存脱敏的新旧值 JSON，便于重现变更。它是只追加事件记录，不作为业务表真值来源。

上述反范式均具有明确的历史一致性或审计目的，不用于规避合理关系建模。

## 6. 实体完整性

- 每张业务表均有非空主键。
- 主键统一使用无符号整数自增；业务编号使用唯一 UUID。
- `roles` 等固定字典表使用稳定编号和唯一代码。
- 购物车、用户角色等多对多事实通过唯一键防止重复。

## 7. 参照完整性

| 子表关系 | 外键策略 | 原因 |
|---|---|---|
| user_roles → users/roles | RESTRICT | 用户和角色不物理删除 |
| addresses → users | RESTRICT | 用户使用禁用而非删除 |
| merchant_applications → users | RESTRICT | 保留审核事实 |
| shops → users | RESTRICT | 保留店主归属 |
| products → shops/categories | RESTRICT | 已引用对象不可物理删除 |
| cart_items → users/products | CASCADE | 若执行维护性删除，临时购物车可同步清理 |
| master_orders → users | RESTRICT | 订单必须保留买家 |
| master_orders → addresses | RESTRICT | 订单已有地址快照，当前实现不物理删除已被订单引用的地址 |
| shop_orders → master_orders/shops | RESTRICT | 保留交易链路 |
| order_items → shop_orders/products | RESTRICT | 保留交易明细和来源 |
| payments/refunds → orders | RESTRICT | 保留资金状态事实 |
| price_history → products | RESTRICT | 价格历史不可失去主体 |
| audit_logs → users | SET NULL | 用户禁用或维护删除不破坏日志 |

正常业务不执行用户、店铺、商品和订单物理删除，外键动作主要防御维护操作。

## 8. 域完整性

通过数据类型、非空和 CHECK 约束限制字段值：

- `price >= 0`；
- `stock >= 0`；
- `quantity > 0`；
- `line_amount = unit_price * quantity`；
- 所有金额 `>= 0`；
- 用户名、姓名、店铺名和商品名不能为空字符串；
- 状态字段只能取文档定义值；
- 拒绝申请或退款时 `reject_reason` 必填；
- 批准/拒绝后 `reviewed_by` 和 `reviewed_at` 必填。

跨行或跨表条件无法由 CHECK 完整表达时，由事务化 Service、存储过程和触发器共同保证。

## 9. 用户定义完整性

### 店主资料归属

`shops.owner_user_id` 唯一；批准开店事务同时初始化店主资料和 OWNER 角色。数据库唯一约束作为资料归属最终防线，不再表示商品经营隔离。

### 一个当前申请

`merchant_applications.user_id` 唯一。拒绝后更新原记录重新提交，不创建多个当前申请。

### 一个购物车商品

`cart_items(user_id, product_id)` 唯一。重复添加采用原子更新数量。

### 一次支付与一次退款

- `payments.master_order_id` 唯一；
- `refund_requests.shop_order_id` 唯一。

唯一约束同时提供并发下的幂等防线。

### 共享商品权限

店主写操作必须同时使用商品 ID 和版本条件，不能只先查询再更新。订单明细中的商品来自平台共享商品池，由结算存储过程校验商品状态、分类状态和库存。

### 金额一致性

- 明细金额 = 成交单价 × 数量；
- 子订单金额 = 所属明细金额之和；
- 总订单金额 = 所属子订单金额之和；
- 支付金额 = 总订单金额；
- 退款金额 = 子订单金额。

结算过程统一计算金额，客户端金额仅供显示，不作为写入依据。

### 状态完整性

状态迁移必须使用带当前状态条件的 UPDATE。例如发货只能更新 `PENDING_SHIPMENT` 子订单。受影响行数为 0 时返回状态冲突，防止并发覆盖。

## 10. 默认地址约束

MySQL 不能直接使用带条件唯一索引表达“每个用户最多一个 `is_default = true`”。Service 在事务中按以下顺序处理：

1. 锁定该用户地址行；
2. 将原默认地址置为 false；
3. 将目标地址置为 true；
4. 提交事务。

集成测试验证并发设置默认地址后仍只有一个默认值。

## 11. 完整性验证清单

- 尝试写入负价格、零数量和非法状态，数据库拒绝；
- 尝试重复用户名、重复购物车项，唯一约束拒绝；
- 尝试删除被订单引用的商品，外键拒绝；
- 修改地址后历史订单快照不变；
- 商品改价后历史订单成交价不变并生成价格历史；
- 并发结算最后一件库存时只有一个事务成功；
- 退款批准后库存、退款、子订单和总订单一致。
