# NovaMall 分阶段开发计划

## 1. 执行原则

每个阶段都是可验证的最小闭环，严格执行：

```text
阶段文档 → 用户确认 → 失败测试 → 最小实现 → 全量验证 → 阶段验收
```

开始阶段前，在 `docs/phases/` 创建阶段规格，至少包含需求、非目标、API、数据变更、实现方案、测试和验收命令。阶段规格未经确认，不写该阶段代码。

当前既定 6 个阶段已完成。阶段 4 之后采用压缩交付路线，优先保证课程要求的 9 项高阶数据库技术真实落地；业务功能收敛为支撑数据库验收和最终演示的最小路径，不扩展完整电商体验。

## 2. 阶段 0：设计与文档基线

### 范围

- README、AGENTS；
- 需求、架构、ER、范式和完整性；
- 高阶数据库技术与实验计划；
- API、安全、UI、测试、部署和课程报告文档；
- 分阶段开发计划。

### 验收

- 文档无 TBD/TODO 和相互冲突；
- Mermaid 图可解析；
- 表名、状态、角色和 API 命名一致；
- 用户确认文档后才能进入阶段 1。

## 3. 阶段 1：基础设施与认证闭环

### 文档先行

创建 `docs/phases/01-foundation-auth.md`，确认 workspace、环境变量、Session、用户表、角色表和认证 API。

### 实现

- pnpm workspace、React、Express、共享类型和 Docker Compose；
- MySQL 迁移机制和基础健康检查；
- users、roles、user_roles、sessions；
- 注册、登录、退出、当前会话；
- scrypt 密码哈希、手机号 AES、CSRF、统一错误和请求编号；
- 三角色路由骨架。

### 最小闭环

会员可注册、登录、查看加密后的个人资料并退出；管理员和店主测试账号能进入各自空后台；越权请求被拒绝。

### 验收

- 数据库中无明文密码和手机号；
- 登录轮换 Session，退出后失效；
- 三角色权限集成测试通过；
- Docker 从空卷启动。

## 4. 阶段 2：商户入驻闭环

### 文档先行

创建 `docs/phases/02-merchant-onboarding.md`，确认申请状态机、审核事务和店铺字段。

### 实现

- merchant_applications、shops；
- 会员提交、查看和拒绝后重提；
- 管理员申请列表、批准和拒绝；
- 批准事务创建店铺并授予 OWNER 角色；
- 审核审计触发器；
- 开店申请和管理页面。

### 最小闭环

普通会员提交申请，管理员批准后，该会员无需重新注册即可成为店主并进入店主后台。

### 验收

- 一个当前申请约束有效；
- 并发重复审核只有一个成功；
- 事务失败不产生孤立店铺或角色；
- 审计日志正确。

## 5. 阶段 3：商品目录闭环

### 文档先行

创建 `docs/phases/03-catalog.md`，确认分类、商品、图片、搜索、索引和页面状态。

### 实现

- categories、products、product_price_history；
- 管理员分类管理；
- 店主商品 CRUD、库存、上下架、归档和图片上传；
- 公开商品列表、分类筛选、详情和中文全文检索；
- 商品价格历史和关键审计触发器；
- 商品查询复合索引与 FULLTEXT ngram 索引。

### 最小闭环

管理员创建分类，店主发布带图片商品，游客或会员按分类和中文关键词找到商品并查看详情。

### 验收

- 店主共同维护平台共享商品；
- 下架和停用分类不出现在公开列表；
- 非图片和超限文件被拒绝；
- 价格变化自动生成历史；
- 保存索引和全文检索初步实验结果。

## 6. 阶段 4：数据库技术核心落地闭环

### 文档先行

创建 `docs/phases/04-database-core.md`，确认 9 项高阶数据库技术的最小业务载体、数据表、SQL 对象、测试用例、证据目录和验收命令。该阶段以数据库专项验收为主，不追求完整购物体验。

### 实现

- 最小订单域：addresses、cart_items、master_orders、shop_orders、order_items、payments、audit_logs；
- `sp_checkout_cart` 存储过程，覆盖输入/输出参数、统一结算、库存扣减、订单快照、购物车清理和 checkoutToken 幂等；
- 商品价格历史触发器和关键审计触发器，覆盖商品、开店审核、订单状态和角色变更；
- 订单明细、有效销量等只读视图；
- 分类商品、共享商品、会员订单、履约订单、审计查询和全文检索索引；
- 结算、支付、取消和库存恢复所需的最小事务服务；
- 窗口函数 Top 10 查询；
- 手机号 AES 加密继续沿用现有安全边界，并补齐订单快照中的密文验证；
- `docs/evidence/database/` 的证据文件骨架和可复制 SQL/命令。

### 最小闭环

会员使用多个商品构造购物车，调用结算后得到一个总订单和履约子订单；库存、金额、订单快照、支付记录、审计日志和视图结果均可由数据库查询验证。

### 验收

- 9 项高阶数据库技术均有迁移、测试和证据文件；
- 存储过程成功、库存不足回滚和 checkoutToken 幂等测试通过；
- 两连接并发争抢最后库存时仅一个成功，最终库存不为负；
- 触发器自动生成价格历史和脱敏审计日志；
- 视图与基础表查询结果一致；
- Top 10 使用窗口函数且并列排序稳定；
- AES 密文、IV、授权解密和错误密钥测试通过；
- 索引和全文检索证据包含真实数据规模、SQL、执行计划和计时。

## 7. 阶段 5：最小业务演示闭环

### 文档先行

已创建 `docs/phases/05-minimal-demo-flow.md`，确认最终演示只覆盖必要页面、API 和状态，不扩展完整订单售后系统。

### 实现

- 会员地址、购物车、结算、订单列表和模拟支付的最小页面；
- 店主共享履约子订单列表和发货按钮；
- 会员确认收货按钮；
- 管理员只读查看审计日志、Top 10 和数据库技术证据入口；
- 后端新增 `/owner/shop-orders/:shopOrderNo/ship`、`/member/shop-orders/:shopOrderNo/confirm` 和 `/member/shop-orders`，用于最小履约主线；
- 简单取消或退款只保留数据库验收需要的最小 API/测试路径，不做完整售后页面；
- 演示数据种子覆盖管理员、会员、店主、分类、商品、订单和数据库证据查询。

### 最小闭环

用户能通过浏览器完成注册/登录、开店审核、商品发布、搜索、加购、统一结算、模拟支付、店主发货、会员确认收货，并在管理员入口查看数据库技术演示数据。

### 验收

- 关键浏览器 E2E 只覆盖最终演示主线；
- API 集成测试覆盖越权、跨会员和重复支付；
- 取消或退款相关测试能证明有效销量、库存恢复和审计规则，不要求完整 UI；
- 前端页面不暴露数据库实验写 SQL 能力；
- 文档明确标注未实现的完整电商能力为非目标。

## 9. 后续非目标

- 不实现完整售后中心、复杂退款页面、优惠券、评价、推荐、客服、物流跟踪和多级运营报表；
- 不扩展店铺隔离运营、批量导入、复杂库存预占和真实支付网关；
- 不为了未来功能提前抽象订单、库存或营销模块；
- 数据库实验允许使用确定性生成数据，不要求通过 UI 手工制造性能规模数据。

## 10. 后续变更规则

- 新需求先判断是否属于新的维护补充范围；
- 所有新需求先判断是否直接服务课程验收或最终演示；不服务则默认推迟或取消；
- 不为后续需求提前抽象；
- 若改变已确认表结构或接口，先更新阶段文档并重新确认；
- 验收失败时只修复相关范围，不并行扩大功能。
