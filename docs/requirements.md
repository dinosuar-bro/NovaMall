# NovaMall 需求规格说明

## 1. 文档目的

本文定义 NovaMall 的业务范围、角色、功能需求、业务规则、非功能需求和明确不做的内容，是后续架构、数据库、API、测试和验收的共同依据。

## 2. 项目背景

NovaMall 是《数据库课程设计》的网上购物系统。系统不仅需要完成基础电商闭环，还要以真实业务场景展示高级数据库技术。当前业务不再按店铺隔离商品经营，所有店主共同维护平台共享商品池。

## 3. 角色与权限

### 3.1 会员

- 注册、登录和退出；
- 维护姓名、加密手机号和多个收货地址；
- 浏览、分类筛选、全文搜索商品；
- 查看商品详情和全平台热销 Top 10；
- 管理购物车并统一结算；
- 模拟支付、查询订单、确认收货；
- 对待发货子订单申请整单退款；
- 提交开店申请并查看审核结果。

所有注册用户默认拥有会员角色。

### 3.2 店主

店主同时保留会员能力，并额外拥有：

- 查看基础店主资料；
- 新增、编辑、上下架和归档平台共享商品；
- 调整平台共享商品库存；
- 查询共享履约子订单并发货；
- 审核待发货子订单的退款申请；
- 查看平台商品价格变更历史。

多个店主账号共同使用同一套商品数据，不按店铺拆分商品归属。

### 3.3 系统管理员

- 管理会员和店主账号状态；
- 审核开店申请；
- 管理平台统一商品分类；
- 只读查询全平台订单；
- 查询审计日志和数据库技术演示数据。

管理员不代替会员下单，也不修改店主商品。

## 4. 功能需求

### FR-01 注册与登录

- 用户使用唯一用户名、密码和手机号注册；系统自动生成展示名。
- 注册密码最短 8 位，必须包含英文大写、小写和数字。
- 密码以不可逆哈希存储，手机号以 AES 密文存储。
- 登录成功后服务端创建 MySQL Session，并通过 HttpOnly Cookie 标识会话。
- 禁用账号不得登录；角色或账号状态变化对后续请求立即生效。

### FR-02 会员资料与地址

- 会员可以查看和修改自己的姓名、手机号。
- 会员可以添加、修改、删除收货地址，并设置一个默认地址。
- 地址包含收件人、省、市、区和详细地址。

### FR-03 开店申请

- 会员填写申请信息后提交开店申请。
- 同一会员同一时间只能有一个待审核申请。
- 管理员批准后，在同一事务中创建店铺并授予店主角色。
- 管理员拒绝时必须填写原因；会员可修改后重新提交。

### FR-04 分类与商品管理

- 分类由管理员统一维护。
- 店主共同管理平台共享商品。
- 商品包含编号、名称、价格、简介、图片路径、库存、分类和状态。
- 商品状态为草稿、上架、下架或归档。
- 已进入历史订单的商品不得物理删除。

### FR-05 商品浏览与查找

- 未登录用户可浏览商品，但购物车、下单和个人功能必须登录。
- 用户可按分类筛选，并按商品名称和简介全文检索。
- 只展示处于上架状态且分类正常的商品。

### FR-06 购物车

- 会员可添加商品、修改数量、删除商品和清空购物车。
- 同一会员和商品在购物车中只有一条记录，重复添加时增加数量。
- 购物车允许同时包含多个商品。
- 加入购物车不锁定库存，最终库存以下单事务为准。

### FR-07 统一结算

- 一次结算创建一个平台总订单。
- 系统创建履约子订单，每个子订单包含对应订单明细。
- 订单明细保存商品名称、图片、成交单价和数量快照。
- 下单在单个事务中校验并锁定库存、创建订单、扣减库存和清空购物车。
- 任一步失败时整个事务回滚。
- 结算使用唯一幂等令牌；相同令牌重试时返回既有订单，不重复扣库存。

### FR-08 模拟支付

- 总订单创建后处于待支付状态。
- 模拟支付在单个事务中将支付记录、总订单和所有子订单更新为已支付/待发货状态。
- 支付操作必须幂等，重复请求不得重复记账。
- 会员可取消待支付总订单，系统恢复库存并关闭全部子订单。

### FR-09 履约与订单查询

- 会员可查看自己的总订单、子订单和明细。
- 店主可查看共享履约子订单。
- 店主对待发货子订单执行发货。
- 会员确认收货后，子订单变为已完成。
- 总订单状态根据所属子订单聚合。

### FR-10 简单退款

- 仅待发货子订单可申请整单退款。
- 每个子订单最多存在一个退款申请。
- 店主可批准或拒绝待发货退款申请。
- 批准后恢复库存、更新退款和订单状态；拒绝后子订单恢复待发货。
- 不处理部分商品退款、退货物流或真实支付渠道退款。

### FR-11 热销 Top 10

- 排名范围为全平台。
- 统计累计有效销售数量。
- 取消和已退款子订单不计入销量。
- 销量相同时按商品编号升序稳定排序。

### FR-12 审计与数据库演示

- 商品价格、库存、订单状态、角色和审核等关键变更写入审计日志。
- 审计记录包含操作者、请求编号、对象、动作、变更前后数据和时间。
- 管理员可查询审计日志，但不得看到密码、密钥或解密后的敏感字段。

## 5. 核心业务规则

- BR-01：每个注册用户自动获得会员角色。
- BR-02：所有店主共享平台商品池，不按店铺隔离商品归属。
- BR-03：只有审核通过的会员可获得店主角色。
- BR-04：店主可操作平台共享商品和共享履约资源，会员资源仍按会员本人隔离。
- BR-05：商品价格和库存不能为负数，购物车数量必须大于零。
- BR-06：订单金额必须等于所属明细金额之和。
- BR-07：订单状态只能按状态机允许的方向迁移。
- BR-08：库存扣减、恢复和订单创建必须具备原子性。
- BR-09：历史订单使用交易快照，不受后续商品修改影响。
- BR-10：Top 10 只统计有效订单明细。

## 6. 非功能需求

### 安全性

- 密码不可逆哈希；手机号 AES 加密；密钥不写入数据库和仓库。
- Cookie 使用 HttpOnly、SameSite，生产环境启用 Secure。
- 所有写接口执行身份、角色和资源归属校验。
- SQL 必须参数化，禁止拼接用户输入。

### 一致性

- 结算、支付、取消、审核开店和退款使用事务。
- 并发结算使用行锁和稳定加锁顺序，防止超卖并降低死锁概率。

### 可维护性

- 前后端均使用严格 TypeScript，禁止 `any`。
- API、数据库迁移和测试均版本化。
- 模块边界以 `docs/architecture.md` 为准。

### 可部署性

- Docker Compose 一键启动前端、后端和 MySQL。
- 图片和数据库使用持久化卷。
- 提供健康检查、演示数据和清晰启动文档。

## 7. 明确不做

- 商品多规格 SKU；
- 优惠券、积分、秒杀、促销；
- 收藏、评价和推荐算法；
- 真实微信或支付宝支付；
- 真实物流接口；
- 部分商品退款、退货物流和平台仲裁；
- 店铺多员工和按店铺隔离的商品经营；
- 微服务、消息队列、Redis 和 Elasticsearch。

## 8. 总体验收标准

- 三类角色权限符合需求，越权请求被拒绝。
- 完成注册到购物、统一结算、支付、发货、确认收货的主流程。
- 完成开店申请到管理员审核的商户流程。
- 完成待发货整单退款的批准与拒绝流程。
- 9 项高阶数据库技术均有 SQL、测试和真实证据。
- Docker Compose 可从空环境启动，全部自动化测试与构建通过。
